# Python 后端迁移审查闭环

> 目的：每个批次完成后，先让 Claude Code 通过 plugin 调用 Codex 做检查。  
> Codex 未通过前，不允许进入下一批。

## 流程

1. Claude Code 完成当前批次实现
2. Claude Code 通过 plugin 触发 Codex 审查
3. Claude Code 等待 Codex 返回结果
4. 如果 Codex 提出问题，Claude Code 按问题修订
5. 修订后再次触发 Codex 审查
6. 直到 Codex 明确通过，才允许进入下一批

## 审查范围

每次审查只看当前批次范围内的变更：

- 文件是否越界
- 接口是否符合当前批次目标
- 测试是否覆盖关键路径
- 是否存在明显设计错误或未闭合流程
- 是否破坏前后端兼容约定

## 审查重点

### 批次 1
- 启动是否成功
- `/api/v1/health` 是否可用

### 批次 2
- ORM 和 Alembic 是否连通
- 用户表是否可创建

### 批次 3
- 登录闭环是否完整
- 返回结构是否兼容现有前端

### 批次 4
- `relationship` 是否真正落库
- 家庭与成员关系是否清晰

### 批次 5
- OCR 是否返回结构化结果
- 库存状态是否一致

### 批次 6
- 站内消息是否真正可用
- 打卡后流程是否闭合

### 批次 7
- AI 返回是否结构化
- 历史是否可追溯

### 批次 8
- 种子是否可复现
- 清理是否过早

## Codex 审查结论格式

Claude Code 在等待结果时，应要求 Codex 只给出以下三类结论之一：

- `通过`
- `有问题，需修改`
- `不通过，禁止进入下一批`

## Claude Code 必须遵守的停止条件

- Codex 没明确通过前，不得进入下一批
- Codex 指出的问题未修复前，不得结束任务
- 如果当前批次修订后仍不过关，继续在当前批次内修订

## 建议给 Codex 的审查问题

> 请审查当前批次的实现是否满足批次目标、是否存在越界修改、是否破坏现有前端兼容、是否有明显设计问题。  
> 只基于当前批次范围给出结论：通过 / 有问题需修改 / 不通过禁止进入下一批。
