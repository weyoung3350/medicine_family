# Python 后端迁移任务调度表

> 说明：这份文档面向执行调度，不是实现细节说明书。  
> 目标是把现有 NestJS 后端迁移为 Python FastAPI 后端，并保持当前 Web / Flutter 客户端可继续使用。

## 1. 调度目标

- 后端改为 Python 实现，便于参赛讲解
- 保持 `packages/backend` 这个边界不变
- 保持 `/api/v1` 接口前缀不变
- 保持 Web 端和移动端的调用方式尽量不变
- 先完成可演示闭环，再补完善性功能

## 2. 调度原则

1. 先基础，后业务，最后收尾
2. 先把可运行骨架做出来，再逐模块迁移
3. 每一批只做一类相邻功能，避免上下文切换
4. 每一批都必须有可验证结果
5. 不拆微服务，不加额外基础设施

## 3. 优先级排序

### P0
- Python 后端骨架
- 数据库连接与模型基础
- 登录认证

### P1
- 家庭、成员、健康档案
- 药品、库存、OCR、上传

### P2
- 服药计划、打卡、提醒、站内消息
- 病历管理
- AI 问诊

### P3
- 附近药店
- 种子数据
- 文档与旧后端清理

## 4. 推荐执行批次

### 第 1 批：运行骨架

目标：
- 让 Python 后端先跑起来
- 先确认健康检查和路由前缀正确

内容：
- FastAPI 项目骨架
- 配置加载
- 基础路由
- 健康检查

验收：
- `GET /api/v1/health` 返回成功
- 后端能独立启动

建议给 Claude Code 的调度方式：
- 只做这一批
- 做完先停，不要提前碰数据库和业务接口

---

### 第 2 批：数据库底座

目标：
- 把 ORM、会话、迁移、最小模型先固定下来

内容：
- PostgreSQL 连接
- SQLAlchemy 基础封装
- Alembic
- 用户表

验收：
- 可以创建数据库表
- 可以完成最小查询

调度要点：
- 这一批结束后再进入认证
- 这一步是后面所有业务模块的前提

---

### 第 3 批：登录与账号

目标：
- 先把参赛演示最需要的“能登录”做出来

内容：
- 注册
- 登录
- JWT
- 当前用户信息

验收：
- 新用户注册后可登录
- Web 登录页继续可用

调度要点：
- 先固定登录响应结构
- 这一步完成后就可以恢复前端联调

---

### 第 4 批：家庭与健康档案

目标：
- 把项目的家庭主线跑通

内容：
- 家庭创建、加入、列表
- 家庭成员维护
- 健康档案
- `relationship` 字段

验收：
- 能创建家庭并关联成员
- 健康档案可读可写

调度要点：
- 这一批要和登录共享同一套用户上下文
- 不要先做药箱，先把家庭结构稳定下来

---

### 第 5 批：药品与库存

目标：
- 完成药箱管理主干

内容：
- 药品增删改查
- 库存管理
- 低库存、临期判断
- OCR 录入
- 文件上传

验收：
- 能录入药品
- 能形成库存状态
- 能从图片入口走到结构化结果

调度要点：
- OCR 结果要保留人工确认入口
- 不要在这一批引入复杂通知

---

### 第 6 批：服药计划与站内消息

目标：
- 把“服药-打卡-提醒”闭环做出来

内容：
- 服药计划
- 时间点
- 打卡、跳过、漏服
- 依从性统计
- 站内信消息中心

验收：
- 能生成当天计划
- 能完成打卡
- 能在系统内看到提醒消息

调度要点：
- 这一批的通知只做站内信
- 不提前接第三方推送

---

### 第 7 批：病历与 AI

目标：
- 让 AI 和病历能力可展示、可解释

内容：
- 病历 CRUD
- 病历 OCR
- AI 问诊
- 图片分析
- 咨询历史
- 结构化输出

验收：
- 能保存咨询记录
- AI 结果能被前端稳定渲染

调度要点：
- 先保证结果结构化，再谈回答质量
- 先可展示，再优化提示词

---

### 第 8 批：药店、种子、清理

目标：
- 收尾成完整演示版本

内容：
- 附近药店查询
- Python 种子脚本
- README 更新
- 旧 NestJS 后端清理

验收：
- 一键种子可跑
- 演示账号可直接登录
- 旧代码移除后项目仍可启动

调度要点：
- 只在全链路验证通过后清理旧后端
- 这一步是发布前最后收尾

## 5. 任务切分建议

### 适合 Claude Code 逐个执行的颗粒度

1. `backend scaffold + health`
2. `db + alembic + user model`
3. `auth`
4. `family + health profile`
5. `medicine + inventory + upload + ocr`
6. `medication + notification center`
7. `medical record + ai`
8. `pharmacy + seed + cleanup`

### 不建议并行的部分

- 数据库底座和认证不要并行改同一批模型
- 药品模块和服药模块不要混在一个任务里
- AI 和病历尽量放在同一批内完成

### 可以并行的部分

- 文档整理
- README 更新
- 种子脚本设计

## 6. 执行门槛

每一批完成后必须满足以下条件才能进入下一批：

- 相关测试通过
- 启动方式明确
- 返回字段稳定
- 前端不需要立刻改代码才能继续演示

## 7. 给执行方的要求

- 每次只领一个批次
- 做完后暂停，等调度确认再继续
- 不要自行扩展到下一批
- 保留变更最小化
- 每批结束时输出：
  - 改了哪些文件
  - 跑了哪些验证
  - 哪些接口已经稳定
