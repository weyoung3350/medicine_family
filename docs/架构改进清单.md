# 家庭健康管家架构改进清单

## 1. 改进目标
- 让参赛作品更容易理解、讲解和演示
- 降低后端实现复杂度，提升可读性和可维护性
- 保留当前产品的核心能力：家庭管理、药箱库存、服药闭环、AI 问诊、适老化体验
- 让演示链路尽量“一键启动、一键种子、一键展示”

## 2. 架构调整方向

### 2.1 后端改为 Python 实现
建议将后端从 NestJS 改为 Python 技术栈。

推荐方案：
- Web API：FastAPI
- ORM：SQLAlchemy 或 SQLModel
- 数据校验：Pydantic
- 数据迁移：Alembic
- 定时任务：APScheduler
- AI 接入：OpenAI SDK / 千问兼容接口
- 文件上传：FastAPI StaticFiles + 本地 uploads

这样做的原因：
- Python 对参赛选手更容易讲解
- FastAPI 的接口结构清晰，学习成本低
- Pydantic 的数据模型适合展示“结构化输入输出”
- Python 生态对 AI 接入更自然

### 2.2 保持模块化单体，不拆微服务
- 后端仍按业务模块划分
- 不引入微服务、消息队列和分布式事务
- 先把模块边界讲清楚，再考虑扩展

推荐模块：
- auth
- family
- medicine
- medication
- medical_record
- ai
- pharmacy
- notification
- scheduler
- upload

### 2.3 前端保持 Web + Flutter 双端
- Web 负责管理和展示完整功能
- Flutter 负责移动端使用场景
- 两端共用同一套接口契约

## 3. 需要优先补强的能力

### P0：Python 后端重写
- 用 FastAPI 重建 API 层
- 复用现有数据库设计思路
- 复用现有业务模块划分
- 保持接口路径风格尽量不变

### P0：演示种子脚本
- 保留并完善一键种子数据
- 启动后直接生成演示账号、家庭、成员、药品、库存、服药计划
- 让演示环境开箱即用

### P0：结构化 AI/OCR 输出
- AI 不只返回聊天文本
- OCR 和问诊结果统一输出 JSON
- 前端按卡片、列表、预警状态渲染
- 保留原始输出，便于回溯和答辩

### P0：站内信消息中心
- 用站内信承接服药提醒、临期提醒、低库存提醒
- 先完成站内闭环，再考虑短信、邮件、第三方推送
- 避免把 App 上架和外部推送作为前置依赖

### P0：适老化大字模式
- 单独设计老人使用入口
- 大字、高对比、少操作
- 服药确认尽量一步完成
- 与普通管理界面分开

## 4. 中优先级改进

### P1：统一数据契约
- 统一成员关系字段
- 统一健康档案字段
- 统一服药计划字段
- 统一 AI/OCR 返回结构

### P1：接口文档和演示文档
- 生成清晰的 API 文档
- 补充“如何启动、如何种子、如何演示”的说明
- 给评委准备一条固定演示路线

### P1：预警与统计更直观
- 药箱首页直接展示低库存和临期
- 服药统计突出依从率和漏服率
- 用最少页面传达最重要的信息

## 5. 低优先级改进

### P2：后续扩展能力
- 短信推送
- 邮件推送
- 第三方通知渠道
- 更复杂的实时同步

### P2：部署和工程整合
- 统一前后端启动脚本
- 将 Flutter 工程纳入更清晰的协作说明
- 如果后续需要，再考虑容器化

## 6. 推荐迁移顺序
1. 明确 Python 后端目标结构
2. 搭建 FastAPI 基础工程和数据库连接
3. 迁移认证、家庭、药品、服药计划主链路
4. 迁移 OCR、AI、病历模块
5. 补站内信和适老化页面
6. 完善 seed 脚本和演示文档

## 7. 结论
如果这是一个高一学生参赛作品，最合理的策略是：
- 后端改成 Python，降低理解门槛
- 继续保持模块化单体，避免复杂化
- 把演示链路、结构化 AI、站内信、适老化模式做扎实

这样更利于讲解，也更利于现场演示和答辩。
